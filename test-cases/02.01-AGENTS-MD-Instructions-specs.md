# Agent Instructions Hierarchy (AGENTS.md specs)

> **Research foundation:** This spec is grounded in the findings of [*Are Agent Instruction Files Necessary with State-of-the-Art LLMs?*](../01-Solution-Vision-Scope-and-Deliverables-Plan/agent-instructions-research-paper.md) вЂ” an empirical investigation combining a literature synthesis of 19 independent studies (1,925+ repositories, 807-project causal analyses, randomized controlled trials) with controlled code-generation experiments. The research directly informs the content guidelines and architectural decisions throughout this spec.
>
> **Core finding вЂ” "less is more":** The research consistently shows that **shorter, constraint-focused instruction files outperform verbose ones**. Overloaded files *reduce* task success rates while increasing inference cost [6]. The optimal AGENTS.md is a short guardrail document (aim for **~50вЂ“80 lines**, never exceeding ~120) that documents what the model must *never* do вЂ” prohibitions, architectural boundaries, and security guardrails вЂ” while letting the code speak for itself on everything else. The research paper's rule surface area analysis found that only 42% of a repository's rules cannot be inferred from code (inferability 4вЂ“5) вЂ” these are the rules worth putting in AGENTS.md. Everything else either goes in on-demand `.agents/instructions/` files or is omitted entirely.

---

## Executive Summary

This spec defines an opinionated approach to agent instructions across all major tools вЂ” VS Code Copilot Chat, GitHub Copilot CLI, and Claude Code вЂ” with a single set of shared instructions and content.

This generalized spec benefits from synergies, discussions, and learnings from E+D teams that informed many of the approaches detailed below for monorepos [Alex Babanov - E+D - Monorepos and agent instructions WORD doc](https://microsoft.sharepoint-df.com/:w:/t/ExcelOPAI/cQqMgxYxb-e6QqYbshi1oSveEgUCnheeqBeiPaQuK7eioQPdMA).

The core idea is simple: **AGENTS.md is the natively-discoverable hub at each project level.** It's read by VS Code Copilot Chat, Copilot CLI, and Claude Code (via `@AGENTS.md` transclusion in CLAUDE.md вЂ” a Claude Code-specific feature). One type of file, multiple tools, zero duplication in instructions.

This spec also aligns with the **emerging `.agents/` ecosystem standard** ([agentskills/agentskills#15](https://github.com/agentskills/agentskills/issues/15)). Specifically, **`.agents/skills/`** has already been adopted as a de facto standard by 15+ tools for cross-tool skill discovery вЂ” including VS Code, OpenAI Codex, Cursor, Windsurf, Gemini CLI, Amp, Goose, RooCode, and others. This spec extends that established `.agents/` pattern by proposing additional subdirectories вЂ” `.agents/instructions/` for deep-dive instructions, `.agents/docs/` for soft-linked reference material, and `.agents/agents/` for custom agent personas вЂ” consolidating all agent customization under a single dot-folder, reducing root-level clutter, and providing cross-tool portability. This broader `.agents/` layout is a proposed direction pending full standardization.

Both single-project and monorepo layouts are first-class for this approach вЂ” see next sections for the detailed patterns.

### Expected Impact (per project)

- **GitHub Copilot in CLI and VSCode** goes from repo-root-managed instructions to nested, collocated sub-folders-level AGENTS.md context
- **Claude Code** gets the same content through `@AGENTS.md` transclusion in CLAUDE.md (Claude Code-specific).

In a monorepo, these gains multiply: each project/domain gets its own discoverable context, and cross-project links let agents follow references across boundaries on demand.

### Why, When, and What to Put in AGENTS.md

**Why:** Instruction files work вЂ” agents faithfully follow them вЂ” but **verbose files hurt performance**. An independent evaluation across multiple agents found that overloaded instruction files *reduce* task success rates while increasing inference cost. Both Cursor (1,000 agents) and Anthropic (16 agents) independently converged on "constraints over instructions" as a universal principle: negative constraints ("no TODOs, no partial implementations") consistently outperform prescriptive task lists.

**When:** The highest-value content is what the model **cannot discover from code on its own**:

- **Prohibitions and negative rules** ("never use this API," "don't ever do THIS in this layer") вЂ” these produced the largest quality gains in experiments and are, by definition, invisible in code
- **Architectural boundary enforcements** вЂ” the model cannot distinguish "we don't have I/O here" from "we *must never* add I/O here"
- **Security and lifecycle guardrails** вЂ” credential delegation policies, resource cleanup mandates

Conversely, project overviews, technology stack descriptions, and naming conventions that are already visible in code should be **omitted** вЂ” they add noise that degrades agent output quality without providing information the model needs.

**What to write:** Engineering teams should treat AGENTS.md as **short guardrail documents** (aim for ~50вЂ“80 lines, never exceeding ~120), not comprehensive READMEs. Shorter is better вЂ” a focused 50-line file of negative rules outperforms a verbose 120-line file that mixes guardrails with inferable content. Document what the model must *never* do. Let the code speak for itself on everything else. AGENTS.md supports nested/scoped placement in sub-folders, domains, and projects вЂ” especially important for monorepos where different areas have different critical rules.

> **вљ пёЏ Required VS Code setting:** Nested AGENTS.md files in subfolders require `chat.useNestedAgentsMdFiles: true` in your repo's `.vscode/settings.json`. Root AGENTS.md is auto-loaded by default (GA, `chat.useAgentsMdFile: true`), but nested discovery is still experimental and off by default in VS Code. Copilot CLI supports nested AGENTS.md natively in GA вЂ” no setting needed. Include this in your repo:
>
> ```json
> // .vscode/settings.json
> {
>   "chat.useNestedAgentsMdFiles": true
> }
> ```

## 1. Target Hierarchy

### 1.1 File Types at a Glance

| File | Purpose | Required? | Audience |
|------|---------|-----------|----------|
| **AGENTS.md** | Hub: guardrails, constraints, negative rules вЂ” auto-loaded | **Yes** вЂ” at every project level | AI agents (all tools) |
| **CLAUDE.md** | Thin pointer: `@AGENTS.md` + Claude-specific behaviors | **Future** | Claude Code users |
| **.agents/instructions/ files** | Deep-dive instructions: guardrails, boundary rules, build constraints, workflow instructions (e.g., collaboration rules) | **Optional** вЂ” soft-linked from AGENTS.md | AI agents + developers |
| **.agents/docs/ files** | Design rationale, architecture, reference context вЂ” including dual-purpose docs (e.g., `artifact-name.md`) that answer "why was this designed this way?" | **Optional** вЂ” soft-linked from AGENTS.md | Developers + AI agents |
| **.agents/skills/** | Agent Skills: task-specific capabilities and workflows | **Recommended** вЂ” at repo root | AI agents (GitHub Copilot, Codex, Cursor, 15+ tools) |
| **.github/agents/** | Custom agent personas with specialized roles | **Optional** вЂ” at repo root | AI agents (GitHub Copilot) |
| **.agents/agents/** | *(TBD)* Cross-tool agent personas вЂ” no tools scan this path yet | **TBD** | AI agents (cross-tool) |
| **README.md** | Setup, getting started, API usage вЂ” targeting artifact *users* | **Optional** вЂ” at artifact root, when users need onboarding beyond the code | Artifact users, GitHub/ADO folder view |
| **Soft links** | Regular markdown links between files and across projects | **Yes** вЂ” for progressive disclosure | AI agents following references |

> **Principle: let the code speak for itself.** README.md and `.agents/docs/` files are added only where they provide value the code is lacking вЂ” architecture diagrams, design rationale, rejected alternatives, onboarding guides. When the code is self-explanatory, skip these docs.

> **рџљЁ Initial Implementation Scope:** For the first release of the AI-First Dev Starter Pack, we will implement **AGENTS.md** files, **.agents/instructions/ files**, **.agents/docs/ files** (including dual-purpose docs), **.agents/skills/** (at repo root), and **.github/agents/** only. Nested `.agents/skills/` for monorepos is on the roadmap (Codex supports it today; GHCP pending). `.agents/agents/` is TBD pending tool convergence. **CLAUDE.md** files are documented here for completeness and future compatibility but are **not included** in the initial implementation.

At the **repo root**, `.agents/` also contains `skills/` вЂ” supported by GitHub Copilot ([VS Code docs](https://code.visualstudio.com/docs/copilot/customization/agent-skills)) and OpenAI Codex at the workspace root level. Custom agents remain in `.github/agents/` (see section 1.3). At nested levels, `.agents/` contains `instructions/` for deep-dive instructions and optionally `docs/` for soft-linked reference material.

> **Roadmap: nested `.agents/skills/` for monorepos.** OpenAI Codex already scans `.agents/skills/` in every directory from CWD up to the repo root вЂ” explicitly designed for skills scoped to a microservice or module. Once GitHub Copilot adds nested `.agents/skills/` scanning, monorepos will benefit from scoped skill discovery per product/domain. Custom agents in `.agents/agents/` is TBD вЂ” no tools currently scan this path, so `.github/agents/` remains the only option for now.

**AGENTS.md** is the single source of truth for "what does an AI agent need to know to work here?" It follows the [AGENTS.md specification](https://agents.md) and targets ~50вЂ“80 lines (never exceeding ~120) вЂ” enough context to orient an agent, with links to deeper content. Shorter is better: research shows verbose files degrade agent performance [6].

**.agents/instructions/ files** hold detailed instructions that expand on the guardrails in AGENTS.md вЂ” build constraints, dependency rules, prohibited APIs, coding guidelines вЂ” as well as **workflow instructions** that agents must follow when applicable (e.g., collaboration rules, session management protocols). They are loaded on-demand when an agent follows a soft link from AGENTS.md, keeping the hub lean while providing depth when needed.

**.agents/docs/ files** hold design rationale, architecture overviews, reference context, and other supplementary information вЂ” including **dual-purpose docs** (e.g., `example-cli.md`, `example-web-app.md`) that answer *why was this designed this way?*. Unlike instructions (which are rules to follow), docs provide context to inform decisions. They are named after their artifact, written for humans first but invaluable for AI, and discoverable via soft links from AGENTS.md. Optional and loaded on-demand.

This building block repeats at every level of the hierarchy вЂ” the same pattern scales from a single project to a large monorepo. Root AGENTS.md is the entry point for both scenarios. See sections 1.4 and 1.5.

### 1.3 Skills and Custom Agents

Skills and custom agents are **separate from the AGENTS.md instruction hierarchy** вЂ” they are task-specific capabilities and agent personas, not project documentation.

**Current implementation:** Skills live in `.agents/skills/` at the repo root вЂ” the cross-tool standard location already supported by GitHub Copilot in VS Code ([VS Code docs](https://code.visualstudio.com/docs/copilot/customization/agent-skills)), OpenAI Codex, Cursor, Windsurf, Gemini CLI, Amp, and 15+ other tools. Custom agents remain in `.github/agents/` вЂ” the only location GitHub Copilot scans for agent personas today.

| Extension | Current Location | Purpose |
|-----------|-----------------|--------|
| **Skills** | `.agents/skills/` (repo root) | On-demand, task-specific workflows (e.g., "create a feature spec," "add a changelog entry"). Supported by GHCP, Codex, Cursor, and 15+ tools. |
| **Custom Agents** | `.github/agents/` | Agent personas with specialized roles (e.g., "Coder," "Reviewer," "PM"). Only location GHCP scans. |

**Roadmap вЂ” nested `.agents/skills/` and `.agents/agents/`:**

- **Nested `.agents/skills/` for monorepos:** OpenAI Codex already scans `.agents/skills/` in every directory from CWD up to the repo root вЂ” explicitly designed for scoped skills per microservice or module. Once GitHub Copilot adds nested `.agents/skills/` scanning, monorepos will benefit from scoped skill discovery per product/domain.
- **Agents в†’ `.agents/agents/`:** This is TBD. No tools currently scan `.agents/agents/`, so `.github/agents/` remains the only viable location. We will revisit once the ecosystem converges on agent persona discovery within `.agents/`.

> **`.agents/instructions/` and `.agents/docs/` are this spec's proposed extensions** вЂ” building on the established `.agents/` directory. Skills already live in `.agents/skills/` at root. `.agents/agents/` is TBD.

> **Scope:** At the repo root, `.agents/` contains `skills/`, and optionally `instructions/` and `docs/`. Nested folders use `.agents/instructions/` and/or `.agents/docs/` as needed, paired with their local AGENTS.md. Nested `.agents/skills/` is a roadmap item for monorepos (Codex supports it now; GHCP pending). `.agents/agents/` is TBD вЂ” `.github/agents/` remains the only option until tools converge.

Skills and agents are loaded and used by GitHub Copilot (or other agent), on demand, not by the AGENTS.md instruction files. Copilot CLI discovers skill metadata at session start but **lazy-loads the skill body only on invocation** вЂ” keeping auto-loaded context lean.

- **Test projects** have their own AGENTS.md files вЂ” unit tests and integration tests have different conventions and constraints

#### Why no `.github/copilot-instructions.md`?

Root AGENTS.md fully replaces `.github/copilot-instructions.md` with zero gaps вЂ” for **both single-product repos and monorepos**. Both are auto-loaded at the same priority level ("Repository instructions"), so having both creates redundant content that risks drift and violates the "less is more" principle. AGENTS.md is also cross-tool вЂ” it works with Codex, Claude Code, Cursor, and 15+ other tools, not just Copilot.

| Capability | Root `AGENTS.md` | `.github/copilot-instructions.md` | Source |
|-----------|-------------------|-----------------------------------|--------|
| VS Code auto-load | **Yes** (GA, `chat.useAgentsMdFile: true` by default) | **Yes** | [VS Code settings reference](https://code.visualstudio.com/docs/copilot/copilot-settings) |
| Copilot CLI auto-load | **Yes** (native вЂ” repo root as "primary" + CWD as "additional") | **Yes** (from repo root) | [Copilot CLI docs](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions) |
| OpenAI Codex | **Yes** (native) | Not used | [Codex docs](https://developers.openai.com/codex/guides/agents-md) |
| Claude Code | Via `@AGENTS.md` in CLAUDE.md | Not used | [Claude Code docs](https://code.claude.com/docs/en) |
| Cursor, Windsurf, Amp, etc. | **Yes** | Not used | [agentskills.io](https://agentskills.io/) |
| Priority level | "Repository instructions" | "Repository instructions" (same) | [VS Code custom instructions](https://code.visualstudio.com/docs/copilot/customization/custom-instructions) |
| Cross-tool | **15+ tools** | Copilot only | вЂ” |

**Conclusion:** For single-product repos, `.github/copilot-instructions.md` is redundant. AGENTS.md provides identical Copilot coverage plus cross-tool portability. **Monorepos also do not need `.github/copilot-instructions.md`** вЂ” root AGENTS.md has identical behavior in all tools. Copilot CLI explicitly finds the repo root AGENTS.md from any CWD and treats it as "primary instructions" ([source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)). VS Code loads it when workspace root = repo root (the standard workflow).

> **Large monorepo note:** When developers open a subfolder as their VS Code workspace, **neither** root AGENTS.md **nor** `.github/copilot-instructions.md` is loaded вЂ” both are workspace-root-dependent. AGENTS.md's unique advantage is per-folder placement: each subfolder can have its own AGENTS.md that becomes the workspace-root hub. See section 1.5.4 for the full analysis and mitigations.

**Dual-purpose docs (artifact-name.md)** are optional, human-readable documentation files named after the artifact they describe (e.g., `example-cli.md`, `example-web-app.md`). They live in `.agents/docs/` and answer *why was this designed this way?* вЂ” capturing design rationale, architecture decisions, rejected alternatives, and constraints. The "dual" in the name means they serve both developers and AI agents вЂ” written for humans first, but linked from AGENTS.md so agents can discover them when making non-trivial changes. Add them only where the code alone doesn't explain the *why*.

**README.md** files are separate from dual-purpose docs. README.md targets **users** of the artifact вЂ” setup, getting started, API usage, configuration. It lives at the artifact root (visible in GitHub/ADO folder views). Dual-purpose docs target **developers** working on the artifact вЂ” design rationale, architecture, constraints. When both exist, README.md answers "how to use" and `artifact-name.md` answers "why it's built this way."

### 1.5 Monorepo / Multi-Product Repository

For repositories containing multiple products (or domains, sub-domains, etc.), the pattern repeats at every level вЂ” **AGENTS.md at each folder level acts as the hub for that scope**. The hierarchy can be as deep as needed: repo в†’ domain в†’ product в†’ artifact, or any other structure that fits the organization.

> **Important:**
> - **Root AGENTS.md** is always loaded by VS Code (when workspace root = repo root) and Copilot CLI (repo root via git). Root AGENTS.md replaces `.github/copilot-instructions.md` вЂ” see section 1.4 for the full comparison.
> - **вљ пёЏ VS Code subfolder workspace gap (large monorepos):** When a developer opens a **subfolder** as the VS Code workspace (not the repo root), **no repo-root instruction file is loaded** вЂ” not root AGENTS.md, not `.github/copilot-instructions.md`, not `.github/instructions/`. All are workspace-root-relative. **The mitigation is AGENTS.md at every level developers open as a workspace** вЂ” see section 1.5.4 for the full analysis, tool behavior matrix, and recommended mitigations.
> - **Copilot CLI finds the repo root** via git regardless of CWD вЂ” root AGENTS.md is always loaded as "primary instructions", plus any AGENTS.md in the current working directory is loaded as "additional instructions" ([source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)).
> - **Claude Code** reads CLAUDE.md via hierarchy walk but **can get flaky with deep nesting** вЂ” community signals confirm it doesn't reliably walk deeply nested folders. Keep CLAUDE.md transclusions shallow.
> - **Each AGENTS.md focuses on its own scope** вЂ” artifact-specific guardrails and constraints. Root-level context (repo-wide rules, product routing) is in the root AGENTS.md, so artifact AGENTS.md files do NOT need to duplicate root content. Include soft links to parent hubs for agents that want to follow references deeper.

#### 1.5.2 Linking Conventions

All references between files вЂ” whether within a project (AGENTS.md в†’ .agents/instructions/ files, AGENTS.md в†’ dual-purpose docs) or across projects вЂ” use **markdown hyperlinks** (soft links) that agents can follow on demand. This preserves single source of truth while enabling progressive disclosure.

> **Three types of file references in instruction files:** AI tools treat references differently depending on their syntax. Understanding this spectrum helps control context loading:
>
> | Type | Syntax example | Loading behavior |
> |------|---------------|-----------------|
> | **Plain-text path** | `See .agents/instructions/rules.md for details.` | Least eager вЂ” tools treat this as regular text. The agent *can* read the file if it decides to, but nothing triggers loading. Lowest coupling, lowest discoverability. |
> | **Markdown hyperlink** ("soft link") | `[Rules](.agents/instructions/rules.md)` | On-demand by default вЂ” agents recognize the structured link and can follow it when relevant. VS Code has a setting `chat.includeReferencedInstructions` (default: **false**) that, when enabled, eagerly pulls in referenced instruction files ([source](https://code.visualstudio.com/docs/copilot/copilot-settings)). With the default off, hyperlinks behave like plain-text paths from a loading perspective вЂ” the agent must deliberately choose to follow them. |
> | **Transclusion (`@` import)** | `@.agents/instructions/rules.md` | **Claude Code-only in instruction files** вЂ” Claude Code expands `@` references at launch, inlining the file content into context ([source](https://code.claude.com/docs/en/memory)). Max depth: 5 hops. Copilot (VS Code + CLI) does not support `@path` as an instruction-file include directive; `@` has other meanings in Copilot chat UIs (attaching files, participants, repos), so don't rely on it for transclusion in AGENTS.md or CLAUDE.md content consumed by Copilot. |
>
> **This spec recommends markdown hyperlinks** (soft links) as the default for all cross-references in AGENTS.md. Across all three tools, soft links are **on-demand by default** вЂ” the agent sees the link but only follows it when the task requires it, keeping auto-loaded context lean:
>
> | Behavior | VS Code Copilot Chat | Copilot CLI | Claude Code |
> |----------|---------------------|-------------|-------------|
> | **Soft links (`[text](path)`) in AGENTS.md** | On-demand вЂ” agent follows via file tools | On-demand вЂ” agent follows via file tools | On-demand вЂ” agent follows via file tools |
> | **Setting to make links eager** | `chat.includeReferencedInstructions` (default: **false**) вЂ” when enabled, eagerly pulls in referenced `.instructions.md` files ([source](https://code.visualstudio.com/docs/copilot/copilot-settings)) | No equivalent setting вЂ” links are always on-demand | No equivalent setting вЂ” use `@` transclusion for eager loading ([source](https://code.claude.com/docs/en/memory)) |
> | **Transclusion (`@path`)** | Not supported in instruction files (`@` has other meanings in chat UI) | Not supported in instruction files | **Eagerly loaded** at launch вЂ” inlines file content into context. Max 5 hops. |
>
> Use transclusions only inside CLAUDE.md pointers (Claude Code-specific, not in AGENTS.md). **If your team uses only Copilot (VS Code + CLI), transclusion does not apply to you** вЂ” soft links are the only reference mechanism you need.

**Rules:**

1. **Link to AGENTS.md first, not internal files** вЂ” when referencing another project, link to its AGENTS.md hub; link directly to .agents/instructions/ or dual-purpose docs only when the reference is specific enough to skip the hub
2. **Use relative paths** вЂ” `../../example-class-library/AGENTS.md` not absolute paths
3. **Describe what the reader will find** вЂ” link text should set expectations for what's at the other end
4. **Group related links** вЂ” use a "Related Projects," "See Also," or "Deep-Dive Reference" section rather than scattering links throughout the file
5. **Never `@`-transclude across projects** вЂ” Claude Code-specific; creates hard coupling and inflates auto-load budgets (not applicable if using only Copilot)
6. **AGENTS.md links to .agents/instructions/ and dual-purpose docs** вЂ” the hub is the discovery mechanism; agents follow links from AGENTS.md to reach deeper content

**Why soft links, not transclusions** (relevant only if your team also uses Claude Code)**:**

- `@` transclusions are a Claude Code-specific feature вЂ” they create hard coupling by inlining content into context at launch
- Soft links let agents decide whether to follow based on the task at hand
- Referenced content is on-demand material, not essential session context
- Each fileвЂ™s auto-load budget stays independent

**Example: AGENTS.md linking to its own .agents/instructions/ files and dual-purpose docs**

```markdown

- [Frontend build constraints](.agents/instructions/frontend-build.md) вЂ” build tooling rules, bundler config
- [Dependency rules](.agents/instructions/dependency-rules.md) вЂ” prohibited packages, credential patterns
- [Web app architecture](.agents/docs/example-web-app.md) вЂ” design rationale and key decisions
```

**Example: cross-artifact links in `src/example-web-app/AGENTS.md`**

```markdown

- [Class library](../example-class-library/AGENTS.md) вЂ” shared API contracts
- [Coding guidelines](../../.agents/instructions/coding-guidelines.md) вЂ” repo-wide standards
```

**Example: cross-artifact links in `src/example-class-library/.agents/instructions/api-boundary-rules.md`**

```markdown

- Web app: see [dependency-rules.md](../../example-web-app/.agents/instructions/dependency-rules.md)
```

This is **progressive disclosure**: auto-loaded content is small (~100вЂ“200 lines per project), agents follow links deeper only when needed, and knowledge lives in exactly one place.

#### 1.5.4 Large Monorepo Considerations

In large monorepos (e.g., Office, Windows, Azure), developers typically **open a subfolder** as their VS Code workspace вЂ” not the repo root. For example, a developer working on `product-A/` opens that folder directly. This creates a discovery gap specific to VS Code Copilot Chat that teams must understand and mitigate.

##### The VS Code Subfolder Workspace Gap

When a developer opens a subfolder as their VS Code workspace, **all repo-root instruction files are invisible to VS Code** вЂ” regardless of file type. This is a VS Code platform limitation, not an AGENTS.md limitation:

- VS Code resolves all instruction file paths **relative to the workspace root** ([source](https://code.visualstudio.com/docs/copilot/copilot-settings)): *"`chat.instructionsFilesLocations`: Relative paths are resolved from the root folder(s) of your workspace."*
- `.github/copilot-instructions.md` must be *"in the `.github` folder at the root of your workspace"* ([source](https://code.visualstudio.com/docs/copilot/customization/custom-instructions))
- `AGENTS.md` is *"automatically detected in the root of your workspace"* ([source](https://code.visualstudio.com/docs/copilot/customization/custom-instructions))
- `.github/instructions/*.instructions.md` is found in the `.github/instructions` folder **of your workspace** ([source](https://code.visualstudio.com/docs/copilot/copilot-settings))

**All three file types have identical behavior** вЂ” none reaches the repo root when VS Code opens a subfolder. This is critical: `.github/copilot-instructions.md` provides **zero additional coverage** over root AGENTS.md in this scenario.

> **Empirically validated (March 2026):** Canary strings placed in repo-root `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md` (with `applyTo: "**/*.md"`), and root `AGENTS.md` were all invisible to VS Code Copilot Chat when a subfolder was opened as workspace. Confirmed: VS Code does NOT walk up to the git root for any instruction file type.

Other tools do NOT have this limitation:

| File at repo root | VS Code Chat (subfolder workspace) | Copilot CLI | Claude Code |
|---|---|---|---|
| `.github/copilot-instructions.md` | **NOT loaded** | **Loaded** (repo root via git) | N/A |
| `.github/instructions/*.instructions.md` | **NOT loaded** | **Loaded** (repo root via git + CWD вЂ” [source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)) | N/A |
| Root `AGENTS.md` | **NOT loaded** | **Loaded** as "primary instructions" (repo root via git вЂ” [source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)) | **Loaded** (hierarchy walk up вЂ” [source](https://code.claude.com/docs/en/memory)) |
| Subfolder's own `AGENTS.md` | **Loaded** (it IS workspace root) | **Loaded** as "additional instructions" | **Loaded** (CWD) |
| Nested AGENTS.md below subfolder | If `useNestedAgentsMdFiles: true` | Not loaded (CWD only) | **Loaded** (on-demand) |

**Key insight:** VS Code Copilot Chat is the **only tool** that loses repo-root context in the subfolder scenario. Copilot CLI and Claude Code both have mechanisms to find the repo root regardless of CWD.

##### Why AGENTS.md Is the Best Mitigation

AGENTS.md has a unique advantage that `.github/copilot-instructions.md` and `.github/instructions/` lack: **it can be placed at every folder level**. When a developer opens `product-A/` as their VS Code workspace:

1. `product-A/AGENTS.md` **IS loaded** вЂ” it's at the workspace root
2. With `chat.useNestedAgentsMdFiles: true`, all AGENTS.md files below `product-A/` are also found
3. Copilot CLI simultaneously loads repo root AGENTS.md as "primary" + `product-A/AGENTS.md` as "additional"

This per-folder placement is impossible with `.github/copilot-instructions.md` (only one per repo at `.github/`) and impractical with `.github/instructions/` (centralized at repo root). AGENTS.md's nested pattern **is** the solution.

##### Recommended Mitigations for Large Monorepos

1. **Place AGENTS.md at every level developers open as a VS Code workspace** вЂ” this is the primary mitigation. If developers typically open `product-A/`, ensure `product-A/AGENTS.md` exists. This file becomes the workspace-root hub for VS Code.

2. **Enable nested AGENTS.md discovery** вЂ” include `chat.useNestedAgentsMdFiles: true` in `.vscode/settings.json` at the repo root AND at each product level (the `.vscode/` folder travels with the workspace).

3. **Use organization-level instructions for cross-repo standards** вЂ” VS Code supports GitHub organization-level instructions that are *"automatically applied to all chat requests"* regardless of workspace root ([source](https://code.visualstudio.com/docs/copilot/customization/custom-instructions)). Use these for org-wide coding standards that must apply everywhere. Enable with `github.copilot.chat.organizationInstructions.enabled: true`.

4. **Consider multi-root workspaces** вЂ” VS Code [multi-root workspaces](https://code.visualstudio.com/docs/editor/multi-root-workspaces) can include both the repo root and a subfolder. This loads root AGENTS.md alongside product AGENTS.md. However, this is a developer workflow change and may not fit all teams.

5. **Use `COPILOT_CUSTOM_INSTRUCTIONS_DIRS` for CLI users** вЂ” Copilot CLI supports a `COPILOT_CUSTOM_INSTRUCTIONS_DIRS` environment variable pointing to additional directories with AGENTS.md files ([source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)). Teams can set this to always include the repo root.

##### Depth Guidance

The hierarchy can be as deep as the repo structure requires вЂ” the AGENTS.md spec explicitly supports unlimited nesting: *"Place another AGENTS.md inside each package. Agents automatically read the nearest file in the directory tree [...] For example, at time of writing the official https://agents.md site, main OpenAI large monorepo has 88 nested AGENTS.md files."* ([source](https://agents.md/#examples))

For **practical monorepos**, we recommend focusing AGENTS.md placement at the levels developers actually open as VS Code workspaces вЂ” typically 2вЂ“3 levels:

```
repo-root/AGENTS.md              в†ђ Level 1: repo-wide routing hub
  product-A/AGENTS.md            в†ђ Level 2: product guardrails (this is typically where VS Code opens)
    src/payments-api/AGENTS.md   в†ђ Level 3: artifact-specific rules
```

Beyond 3 levels, context accumulation and maintenance burden increase without proportional benefit. Deeper rules should be expressed in `.agents/instructions/` files linked from the Level 3 AGENTS.md, not as additional AGENTS.md nesting.

For context budget, the total auto-loaded content across all levels that a tool sees simultaneously should stay under ~200 lines. Each individual AGENTS.md targets ~50вЂ“80 lines (max ~120) per the "less is more" principle.

### 1.6 Crossing the Claude/Copilot Divide

AGENTS.md is the bridge. Root AGENTS.md is natively discovered by all tools (see section 1.4 for comparative analysis). In monorepos, the root AGENTS.md serves as the routing hub; nested AGENTS.md files provide product/artifact-level guardrails.

For **monorepos**, the root-level tool discovery works as follows:

| Layer | Claude Code | Copilot (VS Code + CLI) |
|-------|-------------|-------------------------|
| **Root entry point** | `.claude/CLAUDE.md` в†’ `@AGENTS.md` (always loaded) | Root AGENTS.md (always loaded, GA) |
| **Root hub content** | Via `@AGENTS.md` transclusion | AGENTS.md directly (auto-loaded) |
| **Subdirectory hub** | Via CLAUDE.md `@AGENTS.md` at each level | Nearest AGENTS.md (natively discovered вЂ” no parent walk in VS Code) |
| **.agents/instructions/ files** | On-demand (soft-linked from AGENTS.md) | On-demand (soft-linked from AGENTS.md) |
| **.agents/skills/** | Not natively discovered | Discovered ([VS Code docs](https://code.visualstudio.com/docs/copilot/customization/agent-skills), Codex, Cursor, etc.) |
| **.agents/agents/** | *(TBD)* No tools scan this path yet | *(TBD)* |
| **.github/agents/** | Not natively discovered | Discovered (current location; will move to `.agents/agents/` once GHCP supports it) |

The approaches are complementary: root AGENTS.md is auto-loaded by all Copilot tools (GA); `.claude/CLAUDE.md` transcludes `@AGENTS.md` for Claude Code. Sub-folders use nested `AGENTS.md` because it's natively discovered by Copilot in subdirectories. Both use `.agents/instructions/` for detailed instructions loaded on-demand.

> **вљ пёЏ Tool behavior source:** Discovery behaviors documented here reflect GHCP Product Group guidance (Feb 2026). VS Code parent-folder walking is a known gap being tracked by the product group. Claude Code deep nesting can be flaky. Validate empirically for your repo depth.

### 1.7 Two Documentation Layers

The architecture recognizes two complementary layers of documentation that serve different purposes:

**AI-specific instruction docs** (AGENTS.md, .agents/instructions/) answer *what are the rules here?*, *what must I never do?*, and *how must I work here?*. They're optimized for fast discovery and context compression вЂ” giving an AI agent the guardrails, constraints, and procedural instructions it needs to operate correctly. AGENTS.md carries the critical, code-invisible rules (negative rules, architectural boundaries); .agents/instructions/ files carry detailed instructions loaded on-demand вЂ” both constraint-based (boundary rules, prohibited APIs) and workflow-based (collaboration rules, session protocols).

**Dual-purpose docs (artifact-name.md files)** in `.agents/docs/` answer *why was this designed this way?* They're written for human developers first, but invaluable for AI when making non-trivial changes that require understanding design rationale, rejected alternatives, and architectural constraints.

**README.md** at the artifact root answers *how do I use this?* вЂ” targeting users of the artifact with setup, getting started, and API usage. When both exist, they serve complementary audiences: README.md for users, dual-purpose docs for developers and AI.

```
рџ“Ѓ framework/artifact-A/
в”њв”Ђв”Ђ AGENTS.md                    в†ђ Guardrails & enforcements (must/must-not rules)
в”њв”Ђв”Ђ README.md                    в†ђ (Optional) Setup, getting started вЂ” for users
в””в”Ђв”Ђ рџ“Ѓ .agents/
    в””в”Ђв”Ђ рџ“Ѓ docs/
        в””в”Ђв”Ђ artifact-A.md        в†ђ (Optional) Design rationale, architecture вЂ” for developers + AI
```

How the layers work together:

| Question | Answered by | Example source |
|----------|-------------|----------------|
| "What is this project/artifact?" | AI instructions (AGENTS.md) | Brief description, dependencies, build targets |
| "What must I never do here?" | AI instructions (AGENTS.md + .agents/instructions/) | Negative rules, architectural boundaries, prohibited APIs |
| "How must I work here?" | AI instructions (.agents/instructions/) | Collaboration rules, session protocols |
| "How do I add a dependency?" | AI instructions (.agents/instructions/dependency-rules.md) | Build syntax, placement rules |
| "Why does this use a mediator pattern?" | Dual-purpose doc (artifact-name.md) | Design rationale, alternatives considered |
| "Why was this data structure chosen?" | Dual-purpose doc (artifact-name.md) | Performance analysis, trade-offs |

AI-specific docs compress context and route the agent to the right place quickly. Dual-purpose docs provide the depth needed for changes that go beyond mechanical modifications.

An agent working on an artifact reads AGENTS.md first for orientation, then the artifact's dual-purpose doc (`.agents/docs/artifact-name.md`) when it needs to understand the "why" behind the code. AGENTS.md hubs link to relevant dual-purpose docs, making them discoverable even when the agent doesn't know they exist.

This is not a future aspiration. Dual-purpose docs may already exist in your projects вЂ” possibly as `artifact-name.md` files next to code, or as `README.md` files that mix user-facing and developer-facing content. This architecture gives them a consistent home in `.agents/docs/` and makes them discoverable by linking from AGENTS.md hubs. The spec does not mandate creating dual-purpose docs for all artifacts вЂ” that investment happens organically, driven by the teams that own them.

### 1.8 Core Principles

1. **AGENTS.md is the natively-discoverable entry point at each level** вЂ” read by VS Code Copilot Chat (workspace root + nested subfolder discovery), Copilot CLI (repo root as "primary" + CWD as "additional" вЂ” [source](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions)), and Claude Code (via `@AGENTS.md` transclusion in CLAUDE.md)
2. **Nested hierarchy scales to any depth** вЂ” AGENTS.md at each scope level (repo, domain, product, artifact, etc.); deeper rules go in `.agents/instructions/` to keep each hub lean
3. **CLAUDE.md is a pointer plus Claude-specific behaviors** *(future)* вЂ” uses `@AGENTS.md` transclusion to load the hub content, keeps skills reference and agent config
4. **.agents/instructions/ files hold detailed instructions** вЂ” boundary rules, prohibited APIs, build constraints, coding guidelines, and workflow instructions (collaboration rules, session protocols) вЂ” soft-linked on-demand from AGENTS.md
5. **Unified `.agents/` directory** вЂ” skills in `.agents/skills/` (supported by GHCP + Codex at root), deep-dive instructions in `.agents/instructions/`, reference docs in `.agents/docs/` вЂ” reducing root-level dot-folder clutter. Custom agents in `.github/agents/` for now (will move to `.agents/agents/` once GHCP supports it); `.agents/agents/` is TBD, pending tool convergence
6. **Single source of truth per topic** вЂ” coding guidelines in one .agents/instructions/ file, referenced from anywhere that needs them
7. **Budget-conscious auto-loading** вЂ” AGENTS.md files target ~50вЂ“80 lines (max ~120); shorter is better; .agents/instructions/ files loaded on-demand only
8. **Soft links for all references** вЂ” regular markdown links between files (AGENTS.md в†’ .agents/instructions/, AGENTS.md в†’ dual-purpose docs, cross-project); no `@` transclusions across boundaries
9. **Integrate with repo root conventions** вЂ” follow the root's .agents/instructions/ pattern for deep-dive instructions; adopt AGENTS.md for native Copilot discovery at the sub-project level
10. **Embrace dual-purpose documentation** вЂ” dual-purpose docs (artifact-name.md files in `.agents/docs/`) written for developers are valuable for AI too; link to them from AGENTS.md hubs rather than duplicating their content into AI-specific files

## 2. Content Templates & Conventions

### 2.1 AGENTS.md File Pattern

Every AGENTS.md starts with:

```markdown
# {Project/Artifact Name}

{One-paragraph overview of what this project is and what AI agents need to know.}

## Key Concepts
...
```

This follows the [AGENTS.md specification](https://agents.md) and matches what VS Code Copilot Chat, Copilot CLI, and (via `@AGENTS.md`) Claude Code consume.

### 2.2 .agents/instructions/ File Header Pattern

Every `.agents/instructions/` deep-dive file starts with:

```markdown
# {Topic Title}

**When to read:** {One-sentence trigger condition.}
```

This provides agents with a quick signal of whether the file is relevant to their current task.

### 2.3 Where Does This Content Go?

Use this table to place new AI instruction content. Start from the top вЂ” the first match wins:

| What you're writing | Where it goes | Why |
|---------------------|---------------|-----|
| Repo-wide rules (applies to all products/artifacts) | Root `.agents/instructions/{topic}.md` вЂ” linked from root AGENTS.md | Single source of truth; every product links here |
| Product-wide guardrails, constraints, negative rules | `product/AGENTS.md` | Auto-loaded hub вЂ” agents see it immediately |
| Artifact-specific constraints, boundary rules | `product/src/artifact/AGENTS.md` | Auto-loaded for that artifact |
| Detailed instructions (build, coding, dependencies) | `.agents/instructions/{topic}.md` at the relevant level | On-demand вЂ” keeps AGENTS.md lean |
| Workflow instructions (collaboration rules) | `.agents/instructions/{topic}.md` at the relevant level | On-demand вЂ” agents follow these when applicable |
| Skills (task-specific workflows) | `.agents/skills/{skill-name}/SKILL.md` (repo root) | Cross-tool standard; supported by GHCP ([VS Code docs](https://code.visualstudio.com/docs/copilot/customization/agent-skills)) + Codex. Nested: Codex now, GHCP roadmap |
| Custom agent personas | `.github/agents/{agent-name}/` (will move to `.agents/agents/` once GHCP supports it) | Current GHCP-scanned location; `.agents/agents/` pending tool support |
| Design rationale, architecture, "why" decisions | `.agents/docs/{artifact-name}.md` | Dual-purpose doc вЂ” developers + AI |
| Setup, getting started, onboarding | `README.md` at the relevant level | For artifact users, not AI-specific |
| Claude-only behaviors (skills, agent config) | `CLAUDE.md` at the relevant level *(future)* | Tool-specific вЂ” doesn't belong in AGENTS.md |
| Cross-product conventions (monorepo) | Root AGENTS.md (routing hub) + `.agents/instructions/` for deep-dives | Root AGENTS.md is auto-loaded by all tools |

**Rule of thumb:** If it's a guardrail, constraint, or procedural instruction an agent needs to follow в†’ AGENTS.md or `.agents/instructions/`. If it's a reusable task-specific capability в†’ `.agents/skills/`. If it explains *why* в†’ `.agents/docs/{artifact-name}.md`. If it's tool-specific в†’ CLAUDE.md or `.github/`.

### 2.4 Anti-Patterns (Prohibited)

1. **No self-modification instructions** вЂ” never tell AI to update instruction files
2. **No root-level overwrites** вЂ” tooling must not overwrite the repo root's AGENTS.md
3. **No undiscoverable files** вЂ” do not create `.github/copilot-instructions.md` (redundant with root AGENTS.md) or put instruction files at paths tools can't find
4. **No duplicated content across projects** вЂ” single source of truth per topic; other projects link, not copy
5. **No auto-loading domain-specific content** вЂ” domain-specific content loaded on-demand only
6. **No validation-gate checklists inside AGENTS.md or `.agents/instructions/` instruction files** вЂ” agents ignore markdown checkboxes; instead, express requirements as clear, actionable rules. (Human-only specs/docs like this one may still use checklists for reviewer workflows.)
7. **No tool-specific syntax in AGENTS.md** вЂ” keep AGENTS.md clean of `@` transclusions (Claude Code-specific), Claude skill references, or Copilot-specific directives; tool-specific content belongs in CLAUDE.md or `.github/` files
8. **No `@` transclusions across project boundaries** вЂ” Claude Code-specific; use soft markdown links for cross-project references
9. **No duplicating design rationale into AI docs** вЂ” if a dual-purpose doc (artifact-name.md) explains "why," link to it from AGENTS.md rather than restating it; AI-specific docs cover "what" and "how"
10. **No custom agents in nested `.agents/` folders** вЂ” `.github/agents/` is the only location today (`.agents/agents/` is TBD). Nested `.agents/` directories contain `instructions/` and optionally `docs/`

## 3. Migration Plan

The migration is phased. Each phase is independently valuable вЂ” don't wait for all phases to ship.

#### Phase 1: AGENTS.md Hubs + .agents/instructions/ Deep-Dives

Create AGENTS.md files and .agents/instructions/ deep-dive instructions across **all projects** in the repo. Skills live in `.agents/skills/` at root; custom agents remain in `.github/agents/`:

- **Single-product repo:** Root AGENTS.md + artifact AGENTS.md files + `.agents/instructions/` files at each level + `.agents/skills/` at root + `.github/agents/` at root
- **Monorepo:** Root AGENTS.md (routing hub) + `.agents/skills/` at root + `.github/agents/` at root + product-level AGENTS.md for each product + artifact AGENTS.md files + `.agents/instructions/` files at each level
- **Cross-project links** (monorepo): Add soft markdown links between related products from the start

For each AGENTS.md, include: guardrails, architectural boundaries, negative rules, build commands, and links to `.agents/instructions/` detailed instructions. For `.agents/instructions/` files: coding guidelines, build constraints, dependency rules, boundary rules, and workflow instructions (e.g., collaboration rules).

Validate that VS Code Copilot Chat discovers AGENTS.md at your directory depth (`chat.useNestedAgentsMdFiles: true` must be set in `.vscode/settings.json` вЂ” this setting is experimental and off by default in VS Code; Copilot CLI supports nested AGENTS.md natively in GA).

#### Phase 2: CLAUDE.md Integration *(future вЂ” not in initial implementation)*

> **Deferred.** Teams using Claude Code can add CLAUDE.md thin pointers later:

1. Create CLAUDE.md pointers (`@AGENTS.md` + Claude-specific behaviors) at each project level
2. Validate transclusion behavior вЂ” keep shallow (1вЂ“2 levels) given deep-nesting limitations

#### Migration Principles

1. **No content loss** вЂ” every piece of existing AI content must have a clear destination
2. **No tool regressions** вЂ” tools that currently work must continue to work throughout migration
3. **Validate empirically** вЂ” test tool behavior at each phase, don't assume

## 5. Glossary

| Term | Definition |
|------|-----------|
| **AGENTS.md** | The natively-discoverable AI instruction file at each project level. Read by VS Code Copilot Chat (nested discovery), Copilot CLI (repo root as primary + CWD as additional), and Claude Code (via transclusion). Hub for all tool-agnostic AI context. |
| **CLAUDE.md** | Claude Code's native instruction file. In this architecture, used as a thin pointer (`@AGENTS.md`) plus Claude-specific behaviors. *(Not in initial implementation.)* |
| **.agents/ directory** | Unified directory for all agent customization files. At the repo root, contains `skills/`, `instructions/`, and optionally `docs/`. Custom agents remain in `.github/agents/` (`agents/` within `.agents/` is TBD). At nested levels, contains `instructions/` for deep-dive instructions and optionally `docs/` for reference material. Aligns with the emerging cross-tool `.agents/` standard ([agentskills/agentskills#15](https://github.com/agentskills/agentskills/issues/15)). |
| **.agents/instructions/** | Subdirectory of `.agents/` containing deep-dive instruction files вЂ” both constraint-based (boundary rules, prohibited APIs, build constraints) and workflow-based (collaboration rules, session protocols). Loaded on-demand when agents follow links from AGENTS.md. Replaces the previous `.agent/` convention. |
| **.agents/skills/** | Subdirectory of `.agents/` for Agent Skills (SKILL.md files). Cross-tool standard supported at workspace root by GitHub Copilot ([VS Code docs](https://code.visualstudio.com/docs/copilot/customization/agent-skills)), OpenAI Codex, Cursor, Gemini CLI, and 15+ tools. Nested `.agents/skills/` for monorepos: Codex supports it now; GHCP nested scanning is on the roadmap. |
| **.agents/agents/** | *(TBD)* Subdirectory of `.agents/` for custom agent personas. No tools currently scan this path. `.github/agents/` remains the only option until the ecosystem converges. |
| **artifact-name.md (Dual-purpose doc)** | Human-readable documentation file named after its artifact (e.g., `example-cli.md`), placed in `.agents/docs/`. Contains design rationale, architecture decisions вЂ” written for developers first, but invaluable for AI. "Dual-purpose" refers to the audience: developers + AI agents. Distinct from README.md (which targets artifact *users* with setup/onboarding and lives at the artifact root). Linked from AGENTS.md for AI discoverability. |
| **Hub** | An AGENTS.md file that serves as the entry point for a project or sub-project. Auto-loaded by tools. |
| **Deep-dive** | A .agents/instructions/ file containing detailed reference on a specific topic. Loaded on-demand via links. |
| **Soft link** | A regular markdown link between files or projects. Agents follow on-demand. No auto-loading coupling. |
| **Transclusion** | Claude CodeвЂ™s `@path` syntax that inlines the content of another file at launch. **Claude Code-only in instruction files** вЂ” Copilot (VS Code + CLI) does not support `@` as an include directive in instruction files. Used only inside CLAUDE.md pointers, not in AGENTS.md. |
| **Progressive disclosure** | Pattern where auto-loaded content is minimal; agents follow links to deeper content only when needed. |
| **Context budget** | The total lines of AI instruction content that auto-loads for a given tool and working directory. Target: ~50вЂ“80 lines per AGENTS.md (max ~120). Shorter is better вЂ” research shows verbose files degrade agent performance. |
